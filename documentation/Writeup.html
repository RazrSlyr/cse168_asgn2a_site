Project 2A Writeup: Mixed Reality

Project Description

        The goal of this project was to create a Mixed Reality experience that incorporated real-life spaces and objects with virtual objects. This was done through the creation of a simple game where you create cubes that you can place around and throw at targets.

Goals

        Our plan initially was just to present virtual objects in a mixed-reality environment with some baseline level of interaction. However, thanks to the tools provided by the Unity Engine, we were able to add physics interactions and a simple game.

Results

We’ll go over the different aspects of the project separately.

Representing Real-World Objects in a Virtual Space

        Meta Quest 3 headsets have a feature where they can scan a room to create a “scene”. During this process, the headset can detect where the walls and ceilings are in the space you are in. Optionally, the user can also set boundaries for furniture in the scene. We needed to take this data obtained by the headset and interact with it in the Unity Engine. This is achieved through the use of Meta’s Scene SDK in Unity. We had to provide two different kinds of Prefabs (premade Game Objects) to Meta’s Scene Manager: the Plane Prefab and the Volume Prefab. The Plane Prefab is spawned in for every flat plane like the ceilings, floors, and walls. The Volume Prefab is spawned for every piece of furniture. Since we aren’t currently doing any interactions that utilize any special information about the items in the scene, the prefabs we use are Game Objects that only contain a Collider component.

Spawning in and Interacting With Virtual Cubes

Spawning in the cubes was straightforward using tools Unity provides for creating Game Objects. The code used for the following section can be found mostly in CubeSpawner.cs. The Instantiate function allows us to spawn in a Game Object from a Prefab. The Prefab in this case started as the Unity cube primitive. Inside Unity’s Update function (a function that gets called every frame), we check if the right controller’s index trigger is pressed, and if it is we spawn in this cube at the right controller’s position. We disable the cube’s physics and gravity, as well as make it a child object of the right controller. This is handled This way it will follow the right controller’s movements. When we release the trigger, the cube is released from its parent, and its physics is re-enabled. This is handled in a script we gave to the cube prefab called Grabbable.cs. Specifically, it is handled in the Grab function. This is good for spawning cubes however it isn’t enough for throwing around cubes. For that, we need to take the movement information from our controllers and give that to the cube when we release it. This is handled in the Release function inside of Grabbable. This is, in part, taken from Meta’s Bouncing Ball sample. To be able to re-grab the cubes, we use the Physics.OverlapSphere to find cubes that are near the left controller. From there, the logic is very similar to how we handle the creation of cubes.

Spawning in Targets and Destroying Targets

        The code for this section can be found mostly in TargetSpawner.cs. The function we use to spawn these targets is the same as the cubes (Instantiate), however, there are two main differences. First, we don’t spawn them on a button press but instead on a timer. This is handled by creating a Coroutine that we start at the beginning of the game. This Coroutine, which we’ve called SpawnTarget, waits for 5 seconds before using Instantiate to create a target. This coroutine loops forever. The Instantiation process brings us to the next difference. That being, the spawn position. Instead of spawning it at one of the hands, we need to spawn it somewhere randomly in the scene. The scene, however, is not something that is predefined in the Unity Editor but instead generated by the Meta Quest headset. So, rather than predefined and bound for the spawn position, we have it spawn some random distance away from the center eye position of the headset. While this works for the most part, one drawback is that it does not guarantee that the target spawns in space. The destruction of the target is handled in the OnCollisionEnter function in Target.cs. OnCollisionEnter is a built-in Unity function called the moment any collision happens. If it detects a collision with a cube, it will destroy itself using the Destroy function. Before that, it spawns in the particle effects (using Instantiate) and sets them to self-destruct after they are done (using the Destroy function).

Goals for Part 2

        For Part 2, we aim to make a more interesting game. Specifically, we have two goals. First, is to incorporate Scene Anchor information. The objects that we spawned using Meta’s Scene Manager have Scene Anchors attached to them which store information about what they are in the scene. For example, what is a wall versus what is a ceiling? Currently, we don’t do anything with that information and just treat everything as colliders. We would like to design a game around these anchors. Ideas we were discussing were having a ping-pong-style game about bouncing a ball against a wall for points or having basketball hoops spawn on walls. To achieve either of these though, we would need to use this anchor information to determine where the walls are. The second goal would be to add more of a UI. This would come in the form of a score system for the games.
